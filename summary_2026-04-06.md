# Rodas5 v2 Optimization Summary

## Overview

`solvers/rodas5_v2.py` started as a generic dense Rodas5 ensemble solver and has since been optimized in several stages to reduce JAX tracing overhead, avoid unnecessary automatic differentiation, improve GPU throughput, and add sparse-direct solves through cuDSS.

The current solver is still generic enough to handle:

- nonlinear problems through `f(...)`
- dense linear problems through `jac_fn(...)`
- sparse problems through `jac_sparse_fn(...)`

but it now contains several specialized fast paths when additional structure is available.

## Main Optimizations

### 1. Single-loop batched ensemble execution

The solver uses a single `jax.lax.while_loop` with the batch dimension inside the loop body instead of a pure `vmap` over independent solver loops.

Benefits:

- lower tracing overhead
- better batching of linear algebra
- explicit control of grouping through `batch_size`

`batch_size` controls how many trajectories share one while-loop instance:

- `batch_size=N`: one large grouped solve
- `batch_size=1`: recovers vmap-over-single-problem behavior

### 2. Reusable compiled solver via `make_solver(...)`

The solver now exposes a reusable `make_solver(...)` entrypoint and the convenience wrapper `solve_ensemble(...)` just builds and calls it.

Benefits:

- compile once, reuse many times
- much more accurate benchmarking
- behavior now matches the intended usage pattern of `v7_tc` more closely

### 3. Linear dense fast path through `jac_fn`

For linear systems `dy/dt = J(p) y`, the solver accepts `jac_fn` directly.

This avoids:

- `jax.jacobian(...)`
- `jax.jvp(...)`
- rebuilding the RHS from nonlinear AD machinery

Instead, the solver:

- calls `jac_fn(params)` once per chunk for time/state-independent dense linear systems
- reconstructs `f(y)` as `J @ y`
- reuses the dense Jacobian across all stages and accepted/rejected steps

### 4. Faster GPU matvec path for dense linear systems

The dense linear path uses `lax.dot(...)` instead of generic `einsum(...)`, with TF32-style precision on GPU when `linear_solver_precision="fp32"`.

Benefits:

- better GPU kernel selection
- better throughput for repeated stage matvecs

### 5. FP32 linear solve mode

The solver supports:

- `linear_solver_precision="fp64"` (default)
- `linear_solver_precision="fp32"`

In FP32 mode:

- LU factorization and triangular solves run in `float32`
- surrounding state, adaptivity, save history, and outputs remain `float64`

Benefits:

- lower factorization/solve cost
- better GPU performance while preserving double-precision state/control logic

### 6. Save-time history support

`t_span` is now interpreted as a strictly increasing 1D array of save times, similar to `v7_tc`.

The solver now returns a full history with shape:

`(n_ensemble, n_save, n_vars)`

instead of only the final state.

Benefits:

- direct support for multi-save trajectories
- no need to rerun the solver separately for each output time

### 7. Sparse Jacobian API

The solver now accepts `jac_sparse_fn`.

Supported sparse callback signatures are:

- `(params)`
- `(t, params)`
- `(y, params)`
- `(t, y, params)`

Sparse inputs may be provided as:

- `SparseJacobianCSR`
- JAX `BCOO`
- JAX `BCSR`
- SciPy sparse matrices
- dense arrays, which are converted to CSR

The internal optimized sparse representation is the `SparseJacobianCSR` dataclass:

- static-shape CSR structure
- JAX pytree-compatible
- suitable for traced batched solves

### 8. JAX sparse fallback path

Before cuDSS is used, the solver already has pure-JAX sparse helpers for:

- CSR batched matvec
- CSR to dense conversion

This allows a correctness-preserving dense fallback path whenever cuDSS is unavailable or intentionally bypassed.

### 9. cuDSS sparse-direct solve support through JAX FFI

Sparse direct solves are now integrated through a native cuDSS backend built from:

- `native/rodas5_cudss_ffi.cc`
- `solvers/_cudss_ffi.py`
- `CMakeLists.txt`

The FFI layer:

- builds a shared library in-repo
- registers CUDA custom-call targets with JAX FFI
- creates reusable native solver contexts
- exposes debug counters for analysis, factorization, and solve calls

### 10. Uniform-batch cuDSS execution

The cuDSS path uses uniform-batch mode rather than the earlier non-uniform batch API.

The native context configures:

- `CUDSS_CONFIG_UBATCH_SIZE`
- `CUDSS_CONFIG_UBATCH_INDEX`
- a uniform-batch-safe reordering algorithm

Benefits:

- a single symbolic analysis can be reused across many matrices with the same sparsity pattern
- matches the structure of batched sparse ODE ensembles much better

### 11. Union-pattern sparse batching

For sparse batched solves, the native code constructs a union sparsity pattern across the batch and scatters each trajectory’s matrix values into that shared pattern.

Benefits:

- a single cuDSS analysis per union pattern
- efficient batched factorization/solve across many matrices
- allows batched sparse solves even when values differ across trajectories

### 12. Reuse of factorization across Rodas5 stages

For each step, the solver forms the shifted matrix

`W = I / (dt * gamma) - J`

once, then reuses the resulting factorization across all Rodas5 stage solves for that step.

This is true for:

- dense LU path
- cuDSS sparse-direct path

Benefits:

- avoids refactorizing for each stage RHS
- keeps the sparse path structurally aligned with the dense optimized implementation

### 13. Batched cuDSS support for fixed-structure dynamic sparse callbacks

The changing-pattern sparse support was refined further.

Current behavior:

- if `jac_sparse_fn` is state/time dependent but the CSR structure is fixed and only values change, the solver uses a batched cuDSS path
- if the sparsity structure is genuinely dynamic, the solver falls back away from the batched cuDSS path

This is implemented by:

- probing the initial batch
- checking whether `indptr`, `indices`, and `nnz` match across the batch
- treating that case as a "dynamic values on fixed pattern" path

Benefits:

- preserves batching for important switching/value-dependent sparse systems
- avoids assuming all dynamic sparse callbacks are unsuitable for sparse-direct acceleration

### 14. Separate static and dynamic cuDSS FFI targets

Static sparse solves and dynamic sparse solves use separate JAX FFI target names.

Benefits:

- better isolation of execution patterns at the custom-call level
- reduced risk of JAX/XLA mixing incompatible static and dynamic sparse call shapes

### 15. Native cuDSS context reuse

The Python loader caches cuDSS contexts by:

- `n_vars`
- `batch_size`
- `precision`

Benefits:

- avoids repeatedly creating/destroying native cuDSS handles
- reduces setup overhead
- helps stability across repeated solves and benchmarks

## Features Removed Or Intentionally Not Kept

### Block LU path removed

A blocked LU path similar to `v7_tc` was prototyped for the dense solver, but it did not deliver better performance for `rodas5_v2` and was removed.

The solver now relies on:

- standard dense LU through `jax.scipy.linalg.lu_factor` / `lu_solve`
- cuDSS for sparse-direct solves

## Current Fast Paths

### Dense nonlinear

- input: `f(...)`
- Jacobian obtained through AD
- dense LU solve path

### Dense linear

- input: `jac_fn(...)`
- no AD
- `J @ y` reconstructed directly
- dense LU solve path
- optional FP32 LU

### Sparse linear

- input: `jac_sparse_fn(...)`
- sparse matvec or direct sparse solve depending on backend
- cuDSS on GPU when available
- dense fallback otherwise

### Sparse nonlinear or state/time-dependent sparse Jacobian

- input: `f(...) + jac_sparse_fn(...)`
- `f` provides stage evaluations
- `jac_sparse_fn` provides the linear solve Jacobian
- batched cuDSS is used when the sparse structure is fixed and values vary

## Current Practical Caveats

- `t_span` is now a save-time array, not a `(t0, tf)` pair with final-state-only semantics.
- cuDSS acceleration is GPU-oriented and depends on the native CUDA/cuDSS environment being available.
- The sparse fast path is strongest when the CSR structure is fixed across the batch.
- Fully dynamic sparse structure changes are more difficult to accelerate robustly than fixed-pattern value changes.
- Numerical results for sparse FP32 cuDSS and dense FP32 LU are close but not expected to be bitwise identical. Some sparse tests use slightly looser tolerances for that reason.

## Summary

The main theme of the optimization work has been:

1. reduce JAX overhead
2. avoid unnecessary AD in linear cases
3. improve GPU linear algebra efficiency
4. preserve batching through the whole solve
5. add sparse-direct GPU solves through cuDSS without giving up the generic nonlinear solver structure

At this point, `rodas5_v2.py` is no longer just a generic dense baseline. It is a multi-path solver with:

- reusable compilation
- dense linear and nonlinear modes
- multi-save trajectory output
- FP32 or FP64 linear solve precision
- sparse CSR support
- batched cuDSS acceleration on GPU

while still retaining a dense fallback path for correctness and broader applicability.
